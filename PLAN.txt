Findings

High: the graph timeout is effectively broken, so a “timed out” LLM/graph call can still block the bot for the full underlying runtime. In _invoke_with_timeout(), future.result(timeout=...) is wrapped in a with ThreadPoolExecutor(...). When TimeoutError is raised, the context manager still shuts the executor down with wait=True, which waits for the worker to finish. That means timeout_ms is not a real safety boundary in practice.
High: prompt limiting is still based on the model’s theoretical context window, not the local server’s practical limit, so the code can still send prompts big enough to OOM your MLX server. See MODEL_TOKEN_LIMITS, get_model_token_limit(), and truncate_message(). For qwen-mlx this still treats 262144 as safe, which does not match the real memory ceiling you already hit.
High: two-layer structured conversion has no second-stage size guard. In _ask_llm_once_two_layer(), the first-stage output is turned into first_content_text, then embedded directly into second_prompt without truncation or a dedicated smaller cap. With thinking enabled, the first-stage answer can get long enough to make the conversion pass itself slow or crash-prone.
Medium: startup still eagerly initializes LangMem before any real gameplay decision. In main.py, get_langmem_service() is called immediately after Client(). That triggers local embedding model load in LocalSentenceTransformerEmbeddings and the health-check embedding in _initialize(). For real use, that is still unnecessary startup latency and a timeout risk on cold boot.
